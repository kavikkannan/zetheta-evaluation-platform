# API Design Document

## 1. Overview

This document defines the RESTful API contracts for the distributed candidate evaluation platform. The API layer, exposed through the API Gateway, serves as the unified ingress point for all frontend applications and internal service communication. All endpoints adhere to a consistent request/response envelope, enforce strict input validation, and integrate with the platform's authentication and authorization layers.

## 2. Design Principles

- **Resource-Oriented REST**: Endpoints are modeled around domain entities (`/candidates`, `/submissions`, `/assessments`) rather than actions.
- **JSON Uniformity**: All payloads and responses use JSON with a standard envelope format.
- **Statelessness**: Each request contains all information necessary for authentication and routing; no server-side session state is maintained.
- **Defensive Validation**: All inbound data is validated using Zod schemas before processing.
- **Observability by Design**: Every endpoint emits latency metrics and structured logs via middleware.

## 3. Base URL & Versioning

All routes are prefixed with a version identifier to support backward-compatible evolution:

```
https://api.localhost/v1
```

## 4. Authentication & Authorization

### Authentication Flow

1. The Candidate Portal authenticates a user via the Auth Service.
2. Upon successful credential validation, a long-lived session JWT is issued for portal access.
3. When initiating an assessment, the portal requests a short-lived, single-use cross-application token from the Auth Service.
4. The Assessment Engine validates this token against the Auth Service and Redis nonce store before granting access.

### Authorization Model

Access control is enforced via Role-Based Access Control (RBAC). The API Gateway inspects the user's role and permissions from the JWT payload before forwarding requests to upstream services.

| Role | Permissions |
|------|-------------|
| `candidate` | `read:assessment`, `create:submission` |
| `employer` | `read:dashboard`, `read:scores` |
| `admin` | All employer permissions plus `manage:users`, `manage:roles` |

## 5. Standard Request Format

### Headers

All authenticated requests must include:

```http
Authorization: Bearer <JWT>
Content-Type: application/json
X-Request-ID: <uuid>        # Optional; generated by gateway if omitted
```

### POST / PUT Body Envelope

```json
{
  "data": {
    "field": "value"
  }
}
```

## 6. Standard Response Format

### Success Response (2xx)

```json
{
  "status": "success",
  "data": { }
}
```

### Error Response (4xx / 5xx)

```json
{
  "status": "error",
  "error": {
    "code": "ERROR_CODE",
    "message": "Human-readable description",
    "requestId": "req_uuid"
  }
}
```

### Common Error Codes

| HTTP Status | Error Code | Description |
|-------------|------------|-------------|
| 400 | `VALIDATION_ERROR` | Request body or parameters failed schema validation |
| 401 | `UNAUTHORIZED` | Missing or invalid authentication credentials |
| 403 | `FORBIDDEN` | Authenticated user lacks required permission |
| 404 | `NOT_FOUND` | Requested resource does not exist |
| 409 | `CONFLICT` | Resource conflict (e.g., duplicate submission) |
| 429 | `RATE_LIMITED` | Request quota exceeded |
| 500 | `INTERNAL_ERROR` | Unexpected server failure |

## 7. Endpoint Specifications

### 7.1 Auth

#### POST /v1/auth/login

Authenticates a candidate or employer and returns a session token.

**Request:**

```json
{
  "data": {
    "email": "candidate@example.com",
    "password": "string"
  }
}
```

**Response (200):**

```json
{
  "status": "success",
  "data": {
    "accessToken": "eyJhbGciOiJSUzI1NiIs...",
    "user": {
      "id": "uuid",
      "email": "candidate@example.com",
      "role": "candidate",
      "name": "Jane Doe"
    }
  }
}
```

---

#### POST /v1/auth/token

Generates a short-lived, single-use cross-application token for transitioning from the Candidate Portal to the Assessment Engine.

**Authorization:** Requires valid session JWT.

**Request:**

```json
{
  "data": {
    "applicationId": "uuid"
  }
}
```

**Response (200):**

```json
{
  "status": "success",
  "data": {
    "token": "eyJhbGciOiJSUzI1NiIs...",
    "expiresIn": 60
  }
}
```

---

#### POST /v1/auth/verify

Validates a cross-application token. Used internally by the Assessment Engine.

**Request:**

```json
{
  "data": {
    "token": "eyJhbGciOiJSUzI1NiIs..."
  }
}
```

**Response (200):**

```json
{
  "status": "success",
  "data": {
    "valid": true,
    "payload": {
      "sub": "uuid",
      "email": "candidate@example.com",
      "role": "candidate"
    }
  }
}
```

### 7.2 Assessments

#### GET /v1/assessments/:assessmentId/questions

Retrieves the MCQ question set for an active assessment session.

**Authorization:** `read:assessment`

**Response (200):**

```json
{
  "status": "success",
  "data": {
    "assessmentId": "uuid",
    "questions": [
      {
        "id": "q_uuid",
        "sequence": 1,
        "questionText": "What is the time complexity of binary search?",
        "options": [
          { "label": "A", "text": "O(n)" },
          { "label": "B", "text": "O(log n)" },
          { "label": "C", "text": "O(n²)" }
        ]
      }
    ]
  }
}
```

> **Note:** The correct answer is intentionally omitted from this response to prevent client-side leakage.

### 7.3 Submissions

#### POST /v1/submissions

Submits candidate responses and triggers the asynchronous evaluation pipeline.

**Authorization:** `create:submission`

**Request:**

```json
{
  "data": {
    "applicationId": "uuid",
    "responses": [
      {
        "questionId": "q_uuid",
        "answer": "B"
      }
    ]
  }
}
```

**Response (202 Accepted):**

```json
{
  "status": "success",
  "data": {
    "submissionId": "uuid",
    "message": "Submission received and queued for evaluation"
  }
}
```

### 7.4 Scores

#### GET /v1/scores/:submissionId

Retrieves the computed score and evaluation metadata for a submission.

**Authorization:** `read:scores` (employer owns the candidate) or `candidate` (owns the submission).

**Response (200):**

```json
{
  "status": "success",
  "data": {
    "submissionId": "uuid",
    "score": 85,
    "maxScore": 100,
    "evaluatedAt": "2025-04-16T12:34:56Z",
    "status": "evaluated"
  }
}
```

### 7.5 Dashboard

#### GET /v1/dashboard/candidates

Returns a filterable list of candidates with their current funnel status and scores.

**Authorization:** `read:dashboard`

**Query Parameters:**
- `status` (optional): `applied`, `attempted`, `evaluated`
- `page` (optional): number, default `1`
- `limit` (optional): number, default `20`

**Response (200):**

```json
{
  "status": "success",
  "data": {
    "candidates": [
      {
        "id": "uuid",
        "name": "Jane Doe",
        "email": "candidate@example.com",
        "status": "evaluated",
        "score": 85,
        "evaluatedAt": "2025-04-16T12:34:56Z"
      }
    ],
    "pagination": {
      "page": 1,
      "limit": 20,
      "total": 150
    }
  }
}
```

### 7.6 Admin (RBAC Management)

#### GET /v1/admin/roles

Lists all roles and their associated permissions.

**Authorization:** `manage:roles`

**Response (200):**

```json
{
  "status": "success",
  "data": {
    "roles": [
      {
        "id": "uuid",
        "name": "employer",
        "permissions": ["read:dashboard", "read:scores"]
      }
    ]
  }
}
```

---

#### POST /v1/admin/roles

Creates a new role with a defined permission set.

**Authorization:** `manage:roles`

**Request:**

```json
{
  "data": {
    "name": "senior_reviewer",
    "permissions": ["read:dashboard", "read:scores"]
  }
}
```

**Response (201):**

```json
{
  "status": "success",
  "data": {
    "roleId": "uuid",
    "name": "senior_reviewer"
  }
}
```

## 8. Rate Limiting

To protect service stability and meet latency targets, the API Gateway enforces tiered rate limits:

| Tier | Limit | Scope |
|------|-------|-------|
| Authentication | 10 requests / minute / IP | `/v1/auth/login`, `/v1/auth/token` |
| General API | 100 requests / minute / user | All authenticated endpoints |
| Submissions | 5 requests / minute / user | `POST /v1/submissions` |

Exceeded limits return `429 Too Many Requests` with a `Retry-After` header.

## 9. Validation Strategy

All endpoints validate input through Zod schemas mapped to the request DTOs. Validation failures are caught by a centralized error handler that returns a `400 VALIDATION_ERROR` with a detailed field-level breakdown:

```json
{
  "status": "error",
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Request validation failed",
    "details": [
      { "field": "data.email", "issue": "Invalid email format" }
    ]
  }
}
```

## 10. Design Considerations

- **Consistency**: Every endpoint follows the same envelope structure, simplifying client-side error handling.
- **Security**: Sensitive operations (token generation, score access) are protected by both authentication and RBAC authorization.
- **Performance**: Response payloads are minimal. List endpoints support cursor-based pagination for large datasets.
- **Evolution**: Version prefixing (`/v1`) allows future API iterations without breaking existing clients.

## 11. Conclusion

This API design establishes a secure, consistent, and observable contract between the platform's frontend applications and backend services. By centralizing routing, validation, and authorization in the API Gateway, the system maintains clean service boundaries while delivering a unified developer experience.
